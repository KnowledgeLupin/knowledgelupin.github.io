# 선형대수 독립 포스트 자동화 설계

이 문서는 선형대수 블로그 자동화의 기준 문서입니다.  
목표는 `연재`를 만드는 것이 아니라, 각각이 독립적으로 읽히는 고품질 선형대수 포스트를 하루 두 편씩 안정적으로 생산하는 것입니다.

현재 기준의 핵심은 다음과 같습니다.

1. 공개 글은 연재 번호나 순서를 드러내지 않습니다.
2. 각 글은 엄밀한 수학적 진술과 직접 증명을 포함하는 독립 포스트여야 합니다.
3. 글은 초심 학부생과 머신러닝 때문에 선형대수를 배우는 직장인 모두를 염두에 두고 씁니다.
4. 자동화는 주제를 스스로 고르되, 로컬 자료에 근거한 엄밀성을 확보할 수 있는 경우에만 초안을 생성합니다.
5. 초안 생성 뒤에는 Claude Code 검토만을 통해 문장 흐름과 문체를 최종 교정합니다.
6. 공개 글은 검색엔진에 잘 포착되도록 제목, 설명문, 첫 단락, 태그, slug를 함께 설계합니다.

## 1. 목표 독자와 난이도

핵심 독자는 다음 두 집단입니다.

1. 선형대수를 처음 배우는 학부생
2. 머신러닝을 위해 선형대수를 다시 배우려는 직장인

다음 수준은 기본적으로 가정합니다.

1. 고등학교 수학
2. 미적분학 1의 기초
3. 행렬과 벡터의 기본 표기

다만 이 배경지식을 전제로만 밀어붙이지는 않습니다.  
본문 이해에 꼭 필요한 정의와 사실은 글 안에서 짧게 다시 설명합니다.  
관련 글이 이미 있다면, 필요한 대목에서 본문 중간에만 자연스럽게 링크를 넣습니다.  
관련 글이 아직 없다면, 본문 안에서 필요한 만큼 직접 설명하고 내부 기록에는 `관련 글 작성 필요` 메모를 남깁니다.

## 2. 공개 글의 성격

공개 글은 `연재물의 한 회차`처럼 보이면 안 됩니다.  
독자는 앞선 글을 읽지 않았더라도 지금 읽는 글 하나만으로 핵심 개념, 정리, 증명, 예시, 응용 동기를 얻을 수 있어야 합니다.

따라서 다음 원칙을 둡니다.

1. 제목에는 `1편`, `2편`, `연재`, `다음 글` 같은 표지를 두지 않습니다.
2. 글 끝에 연재 예고를 붙이지 않습니다.
3. 내부적으로는 순서를 관리하지만, 공개 글은 독립 포스트로만 보이게 둡니다.
4. 내부 순서는 기획과 자동화의 편의를 위한 것이고, 공개 질서가 아닙니다.

## 3. 글쓰기 원칙

### 3.1 핵심 방향

각 글은 다음 세 요소를 모두 담아야 합니다.

1. 개념 이해
2. 계산 감각
3. 응용 동기

이 셋은 어느 하나로 대체되지 않습니다.  
단, 전개 순서는 고정하지 않습니다.  
문제 제기에서 시작할 수도 있고, 작은 예제에서 시작할 수도 있고, 응용 상황에서 출발할 수도 있습니다.  
구조 선택은 주제에 따라 자동화가 재량으로 정합니다.

### 3.2 엄밀성

엄밀성은 필수입니다.  
정의, 정리, 증명, 논리 전개는 교과서 수준으로 엄격해야 합니다.  
직관적 설명은 엄밀한 본문을 대신하는 것이 아니라, 그 위에 덧붙는 해설이어야 합니다.
특히 공리나 조건을 소개할 때는 키워드만 나열하지 않고, 각 조건의 의미를 수식과 짧은 해설로 분명히 적습니다.

증명 정책은 다음과 같습니다.

1. 핵심 명제는 가능한 한 직접 증명합니다.
2. 증명이 길어지더라도 원칙적으로 같은 글 안에서 끝까지 제시합니다.
3. 가능한 증명이 여러 개일 때는 `이해하기 쉽고 구조가 잘 드러나는 증명`을 우선합니다.
4. 증명 전략은 길거나 아이디어가 중요한 경우에만 짧게 먼저 제시합니다.

### 3.3 문체

문체는 혼합형으로 둡니다.

1. `정의`, `정리`, `증명`은 단정한 교과서 문체를 씁니다.
2. 주변 설명, 동기, 오해 정리, 응용 해설은 자연스럽고 부드러운 블로그 문체를 씁니다.
3. `정의`, `정리`, `증명` 바깥의 설명 문단은 기본적으로 경어체를 유지합니다.
4. 설명 문단에 반말이나 거친 평서체가 섞이면 안 됩니다.
5. `정리`, `증명` 블록 안의 수학적 서술은 `...라 하자`, `성립한다` 같은 교과서식 평서체를 허용합니다.

직관적 설명은 `해설.` 같은 별도 표지 없이 자연스러운 문단으로 처리합니다.

### 3.4 형식 요소

형식 요소는 다음과 같이 보입니다.

1. `정의`, `정리`, `증명`은 본문과 분리된 블록으로 제시합니다.
2. `정리`에만 번호를 붙입니다.
3. `예제`는 짧은 경우 본문 흐름 안에서 자연스럽게 처리합니다.

## 4. 한 포스트의 품질 기준

각 포스트는 다음 조건을 모두 만족해야 합니다.

1. 핵심 정의가 정확히 제시되어야 합니다.
2. 핵심 정리나 명제가 명시되어야 합니다.
3. 적어도 하나의 직접 증명이 있어야 합니다.
4. 작은 예제를 직접 전개하는 계산이 있어야 합니다.
5. 현실적 응용 예시가 있어야 합니다.
6. 독자가 헷갈릴 지점이 있다면 본문 흐름 안에서 자연스럽게 짚어야 합니다.
7. 앞선 글을 몰라도 읽히도록 필요한 배경을 다시 설명해야 합니다.

자동화는 초안 작성 후 이 일곱 항목의 충족 여부를 자체 점검합니다.

## 5. 포스트의 가시적 구조

공개 글의 가시적 뼈대는 `약한 표준형`으로 둡니다.

1. 서두
2. 본문 소제목 2개 이상
3. 마무리
4. 참고문헌

그 밖의 요소는 주제에 따라 유연하게 둡니다.

1. 짧은 연습문제는 필요할 때만 1-2문항 넣습니다.
2. 코드 단락은 필요할 때만 둡니다.
3. 시각자료 단락은 필요한 주제에서만 둡니다.

## 6. 예제와 응용

각 글은 두 종류의 예시를 모두 비중 있게 다루는 것을 원칙으로 합니다.

1. 작은 예제를 직접 전개하는 계산
2. 현실적인 응용 예시

예를 들어 `직교사영과 최소제곱`이라면, 좌표 공간에서의 작은 계산 예제와 데이터 맞춤 문제를 둘 다 본문 안에서 의미 있게 다룹니다.  
현실 예시는 글 끝에 장식처럼 2-3문장 붙이는 방식으로 처리하지 않습니다.

응용 연결은 기본적으로 넣되, 주제에 따라 머신러닝이 더 자연스러울 수도 있고, 기하, 물리, 최적화가 더 자연스러울 수도 있습니다.  
억지스러운 응용은 피하고, 가장 자연스러운 맥락을 우선합니다.

## 7. 시각자료와 코드

### 7.1 시각자료 사용 원칙

시각자료는 모든 글의 기본 구성요소가 아닙니다.  
필요한 주제에서만 분명하게 사용합니다.

도식의 형식은 다음 원칙을 따릅니다.

1. 기하 그림, 좌표 그림, 선형변환 도식은 `SVG`
2. 데이터 플롯, 산점도, 시뮬레이션 결과는 `PNG`

시각자료가 필요한지, 그리고 그것이 이해에 `핵심적`인지 `보조적`인지는 자동화가 주제마다 판단하고 내부 기록에 이유를 남깁니다.

시각자료를 자동 생성하기 어려우면 다음처럼 처리합니다.

1. 본문은 완성합니다.
2. 그림 자리표시와 제작 메모를 남깁니다.
3. 그림이 핵심 이해에 필수라면 상태는 `needs_figure`에 머뭅니다.
4. 그림이 보조적이라면 `review_ready`로 갈 수 있습니다.

시각자료 캡션은 다음을 모두 포함합니다.

1. 무엇을 보여주는지
2. 왜 중요한지

### 7.2 코드 사용 원칙

Python/NumPy 코드는 필요한 경우에만 짧게 넣습니다.  
코드는 본문 이해를 돕는 수단이지, 모든 글의 정규 요소가 아닙니다.

배치는 주제에 따라 유연하게 고릅니다.

1. 개념 이해에 직접 필요하면 본문 중간에 배치합니다.
2. 수학적 전개가 중심이면 글 끝의 `계산 확인` 성격 단락으로 둡니다.

## 8. 용어와 표기

### 8.1 기본 원칙

1. 용어는 한글 중심으로 씁니다.
2. 영어는 첫 등장에만 병기합니다.
3. 표기법은 가능한 한 `Olver`를 따릅니다.

### 8.2 낯선 표기 처리

`Olver`의 용어가 한국어 초심자나 응용 독자에게 덜 익숙할 수 있습니다.  
이 경우에는 `Olver`의 표기를 유지하되, 첫 등장에 대응 용어와 역할 설명을 함께 적습니다.

예를 들면 다음과 같습니다.

1. `image(상, 흔히 column space라 불리는 부분공간)`
2. `coimage(행공간)`
3. `cokernel(좌영공간)`
4. `pivot(소거 과정의 기준 원소)`

## 9. 제목, slug, 메타데이터

### 9.1 제목

제목은 정확한 개념명 중심으로 둡니다.  
질문형, 홍보형, 연재형 제목은 쓰지 않습니다.

제목을 다듬을 때는 다음 범위까지만 허용합니다.

1. 핵심 개념은 유지합니다.
2. 더 자연스러운 한국어 제목으로 재구성할 수 있습니다.
3. 영어는 고유명사나 널리 굳어진 용어만 유지합니다.
4. 가능하면 실제 검색될 한국어 표현을 제목 안에 직접 포함합니다.
5. 지나치게 추상적인 표현만으로 끝나는 제목은 피합니다.

### 9.2 slug

`slug`는 영어 중심으로 둡니다.  
시각자료 파일명도 포스트 `slug` 기반의 영어 파일명으로 통일합니다.

다만 `slug`는 제목과 검색 의도를 반영해야 합니다.

1. 제목의 핵심 개념과 직접 대응해야 합니다.
2. `scope`, `overview`, `introduction` 같은 일반어만으로 끝나지 않게 합니다.
3. 공개 제목을 크게 바꾸지 않는 한 안정적으로 유지합니다.

### 9.3 front matter

초안의 front matter는 다음 필드를 자동으로 채웁니다.

1. `title`
2. `date`
3. `slug`
4. `categories`
5. `tags`
6. `description`
7. `excerpt`
8. `image`
9. `author`
10. `math`
11. `toc`

세부 원칙은 다음과 같습니다.

1. `categories`는 항상 `Mathematics`, `Linear Algebra`
2. `tags`는 핵심 용어를 자동 추출하되, 검색 의도를 반영한 한국어 표기를 우선합니다.
3. `description`은 제목 반복이 아니라, 글이 무엇을 설명하는지 보여 주는 검색 스니펫이어야 합니다.
4. `excerpt`는 첫 문단 복사본이 아니라, 글의 중심 질문과 독자 효용을 짧게 보여 주는 요약문이어야 합니다.
5. `image`는 시각자료가 있는 글에서 우선 채우고, 없을 때는 사이트 기본 이미지를 사용합니다.
6. `author`는 고정
7. `math: true`
8. `toc: true`

### 9.4 검색 가시성 규칙

검색 최적화는 별도의 홍보 문장을 덧붙이는 일이 아니라, 이미 좋은 글이 검색 의도와 만나는 지점을 분명히 만드는 일입니다.

따라서 각 공개 글은 다음 조건을 기본으로 만족해야 합니다.

1. 제목에는 핵심 개념명과 실제 검색어가 자연스럽게 들어가야 합니다.
2. 첫 두 문단 안에 주 검색어와 그 동의 표현이 무리 없이 등장해야 합니다.
3. 최소 하나의 본문 소제목에도 핵심 검색어가 직접 들어가야 합니다.
4. `description`은 제목과 다른 문장으로 작성해야 합니다.
5. 태그는 개념 분류용 표기뿐 아니라 실제 검색어 표기도 포함해야 합니다.
6. 내부 링크는 독립 포스트 성격을 해치지 않는 범위에서 관련 개념 검색 흐름을 도와야 합니다.
7. 클릭을 유도하기 위한 과장형 제목은 금지합니다.

## 10. 참고자료 정책

### 10.1 자료 우선순위

주자료는 다음 파일로 고정합니다.

- [2018Olver - Applied Linear Algebra.pdf](/Users/kwkwon/Desktop/Obsidian/Mathematician/_shared/my-library/2018Olver%20-%20Applied%20Linear%20Algebra.pdf)

`Olver`로 설명과 응용은 일관되게 끌고 가되, 증명 밀도가 부족한 경우에는 다른 로컬 자료를 보조자료로 씁니다.

보조자료 선택 원칙은 다음과 같습니다.

1. 엄밀한 증명이 가장 잘 갖춰진 로컬 자료를 우선합니다.
2. 보조자료를 쓰더라도 최종 표기와 설명 흐름은 `Olver` 기준으로 통일합니다.

### 10.2 선택 가능 주제

자동화는 로컬 자료가 충분한 주제만 선택합니다.  
엄밀한 진술과 증명을 뒷받침할 수 없으면 해당 주제는 건너뜁니다.

### 10.3 공개 참고문헌

공개 글에는 글 끝에 짧은 `참고문헌`을 둡니다.  
표시는 최소한 다음 정보를 포함합니다.

1. 저자
2. 연도
3. 제목
4. 출판사

가능하면 이 정보는 로컬 `master.bib`를 기준으로 자동 생성합니다.

## 11. 주제 후보와 중복 판정

### 11.1 기본 원칙

핵심 후보 목록은 내부 manifest에 유지합니다.  
자동화는 새로운 후보를 추가할 수 있지만, 다음 조건을 만족할 때만 가능합니다.

1. 주자료나 보조자료에서 `독립된 절`, `독립된 정리 묶음`, `독립된 개념 묶음`으로 분명히 확인될 것

자동화가 새 후보를 발견하면 우선 `title`, `slug`, `source_refs` 정도의 최소 `candidate` 항목만 추가합니다.  
이 새 후보는 사람이 핵심 개념, 대표 정리, 대표 예시를 보강하기 전까지는 실제 집필 대상으로 올리지 않습니다.

manifest는 수학 내용뿐 아니라 검색 설계 정보도 함께 가질 수 있습니다.  
핵심 후보에는 다음 SEO 필드를 둡니다.

1. `primary_query`
2. `secondary_queries`
3. `search_intent`
4. `seo_title`
5. `seo_slug`
6. `meta_description_seed`
7. `related_terms`
8. `faq_queries`
9. `internal_link_targets`
10. `image_hint`

이 정보는 공개 글의 제목, 설명문, 첫 단락, 태그, 내부 링크를 자동 생성하거나 점검하는 기준으로 사용합니다.

### 11.2 같은 주제 판정

`같은 주제`는 제목만으로 판정하지 않습니다.  
다음이 거의 같을 때 같은 주제로 봅니다.

1. 핵심 개념
2. 중심 정리
3. 대표 예시

같은 주제의 공개 글이 이미 있으면 새 글을 만들지 않습니다.  
이미 `published`된 글을 더 좋게 만들고 싶다면, 자동 신규 생성이 아니라 기존 글의 `개정`으로 처리합니다.

부분적으로만 겹치는 주제는 자동화가 독자적으로 판단하지 않습니다.  
내부 기록에 보류 이유를 남기고 다음 후보로 넘어갑니다.

## 12. 주제 선택 점수화

각 후보는 매 실행마다 네 축으로 평가합니다.

1. 중요도
2. 독립성
3. 응용성
4. 검색 적합성

평가값은 `낮음 / 중간 / 높음`의 3점 척도를 쓰고, 이를

$$
\text{낮음}=0,\qquad \text{중간}=1,\qquad \text{높음}=2
$$

로 환산합니다.

총점은 다음 식으로 계산합니다.

$$
\mathrm{score}=3\cdot \mathrm{importance}+3\cdot \mathrm{independence}+2\cdot \mathrm{applicability}+2\cdot \mathrm{searchability}.
$$

검색 적합성은 다음을 함께 봅니다.

1. 실제 검색 가능성이 있는 한국어 질의가 분명한가
2. 제목을 너무 추상적으로 쓰지 않고도 개념 정확성을 유지할 수 있는가
3. 기존 공개 글과 검색 의도가 충돌하지 않는가
4. `description`, `excerpt`, 태그, 소제목을 설계하기 쉬운 주제인가

동점 해소 순서는 다음과 같습니다.

1. 중요도
2. 독립성
3. 검색 적합성
4. 응용성
5. 그래도 같으면 사람 판단

자동화는 이 점수와 간단한 평가 이유를 `검토한 모든 후보`에 대해 기계용 상태 파일에 기록합니다.

## 13. 상태 모델

상태는 다음 아홉 단계로 둡니다.

1. `candidate`
2. `selected`
3. `drafting`
4. `draft_ready`
5. `needs_figure`
6. `needs_revision`
7. `review_ready`
8. `published`
9. `superseded`

추가로 `candidate`에는 `skip_reason`, `last_checked_at`를 기록할 수 있습니다.  
건너뛴 후보는 다음 실행 때마다 다시 평가합니다.

검색 관련 보류는 별도 상태를 늘리지 않고 `candidate` 또는 `needs_revision` 안에서 이유를 기록합니다.  
예를 들어 제목 충돌, 검색 의도 중복, 설명문 설계 실패 같은 정보는 `skip_reason`이나 `failure_reason`에 남깁니다.

## 15. 초안 생성 자동화 흐름

초안 자동화는 하루 두 번 실행합니다.

1. `Europe/Berlin` 기준 `09:00`
2. `Europe/Berlin` 기준 `18:00`

미검토 초안이 쌓여도 멈추지 않습니다.

실행 흐름은 다음과 같습니다.

1. 후보군을 읽습니다.
2. 각 후보의 중요도, 독립성, 응용성, 검색 적합성을 평가합니다.
3. 자료 부족, 부분 중복, 기타 보류 이유가 있으면 기록하고 다음 후보로 넘어갑니다.
4. 가장 적절한 후보를 `selected`로 올립니다.
5. 주자료와 보조자료를 확인합니다.
6. 본문과 함께 제목, `description`, `excerpt`, 태그 초안을 작성합니다.
7. 구조와 내용 기준을 자체 점검합니다.
8. 기준 미달이면 자가 수정을 수행합니다.
9. 자가 수정은 최대 3회까지 허용합니다.
10. 자체 점검을 통과한 초안은 Claude Code 전용 검토 단계로 넘깁니다.
11. Claude Code 검토까지 마치면 `review_ready` 또는 `needs_figure`로 이동합니다.

### 15.1 자체 품질 점검

자동화는 다음을 모두 점검합니다.

1. 구조 기준
2. 품질 기준 1-7
3. 엄밀성
4. 독립 포스트 완결성
5. 응용 예시의 실질성
6. 메타데이터 완결성
7. 제목의 검색 적합성
8. `description`과 `excerpt`의 스니펫 품질
9. 첫 문단과 소제목의 검색어 자연성

### 15.2 실패 처리

최대 3회 수정 후에도 기준을 만족하지 못하면,

1. 초안은 남깁니다.
2. 내부 기록에 실패 사유를 남깁니다.
3. 상태는 `needs_revision`으로 둡니다.
4. 자동화는 다음 후보로 넘어갑니다.

### 15.3 Claude Code 최종 교정

자체 점검을 통과한 초안은 `tools/review-linear-algebra-draft-with-claude.py`로 세 차례 연속 교정합니다.
이 단계의 문장 교정은 Claude Code만 담당하며, Codex가 중간에 문장을 수동으로 고쳐 넣지 않습니다.

Claude Code 검토의 핵심 점검 항목은 다음과 같습니다.

1. 문장과 문단의 흐름이 매끄러운지
2. 단락 전환이 부자연스럽지 않은지
3. 설명 문단의 경어체가 일관되는지
4. `정의`, `정리`, `증명` 밖에서 반말이나 불필요한 평서체가 섞이지 않았는지
5. 소제목이 불필요하게 잘게 나뉘어 있지 않은지
6. 소제목의 이름이 어색하지 않은지
7. 마무리 단락이 글의 논지에 맞게 자연스럽게 닫히는지
8. ChatGPT스럽거나 기계적으로 반복되는 어색한 표현이 남아 있지 않은지

이 단계는 단순 평가가 아니라 실제 초안을 고쳐 쓰는 교정 단계입니다.  
다만 front matter, 수식, 경로, 링크, 참고문헌의 기본 구조는 보존해야 합니다.
Claude Code 검토는 기본적으로 3회 반복합니다.
Claude Code 검토가 실패하거나 멈추면, 자동화는 초안을 수동으로 보정하지 않고 실패 사실만 기록한 뒤 다음 절차로 넘어갑니다.

## 16. 기존 초안 처리

기존 미게시 초안이 있더라도, 다음 경우에는 품질이 낮은 초안으로 간주합니다.

1. 수학적 정확성이나 증명이 부족함
2. 필수 요소가 빠짐
3. 독립 포스트처럼 읽히지 않음
4. 문체나 구조가 현재 기준과 맞지 않음
5. 시각자료, 링크, 메타데이터까지 포함해 전체 완성도가 낮음
6. 제목, 설명문, 태그 때문에 검색 가시성이 현저히 약함

이 경우에는 다음처럼 처리합니다.

1. 기존 초안은 `_drafts/_superseded/`로 옮깁니다.
2. 상태는 `superseded`로 둡니다.
3. 새 초안을 정식 draft 경로에 생성합니다.

## 17. review와 게시

`review_ready` 초안은 사람이 검토한 뒤, 사용자가 직접 `게시`를 명령할 때만 공개로 전환합니다.

게시 명령의 범위는 다음을 포함합니다.

1. `_drafts/`에서 `_posts/`로 이동
2. 날짜 확정
3. front matter 정리
4. 시각자료 경로 점검
5. 링크 점검
6. 로컬 빌드 확인
7. 수식 렌더링 확인
8. 시각자료 누락 여부 확인
9. 빌드된 HTML의 title, description, canonical, `og:image` 점검
10. 제목과 `description`이 검색 스니펫으로 읽혔을 때 자연스러운지 점검
11. `git commit`
12. `git push`

게시는 `main`에 직접 `commit`하고 `push`합니다.  
다만 git에는 공개 산출물과 직접 관련된 파일만 올립니다.

1. `_posts` 파일
2. 관련 이미지
3. 직접 필요한 링크 수정

내부 상태 파일은 게시 커밋에 포함하지 않습니다.

필요하면 게시 직전 개정도 허용합니다.  
이미 생성된 본문이 수학적으로는 충분하더라도, 제목과 설명문이 검색 의도와 맞지 않으면 게시 전에 수정합니다.

게시용 커밋 메시지 형식은 다음을 기본으로 합니다.

`Add post on <slug or topic>`

## 18. published 글의 개정

이미 `published`된 글을 더 좋게 만들고 싶을 때는 새 글을 만들지 않고 기존 글을 개정합니다.

개정 정책은 다음과 같습니다.

1. URL과 `slug`는 유지합니다.
2. 본문은 조용히 수정합니다.
3. 공개적으로 `updated` 메모나 별도 공지는 남기지 않습니다.

## 19. 구현 권고

현 구현은 연재형 manifest와 연재형 생성기를 전제로 한 부분이 있습니다.  
새 설계에 맞는 구현은 다음 구성으로 옮기는 것이 좋습니다.

1. 독립 포스트용 manifest 경로를 새로 둡니다.
2. 연재형 `series` 용어를 `independent posts` 구조로 바꿉니다.
3. 생성기는 제목 순서보다 `후보 평가 -> 선택 -> 작성` 흐름을 갖도록 재작성합니다.
4. 게시용 명령을 별도 스크립트나 별도 automation으로 분리합니다.
5. 생성기와 게시기 모두에서 SEO preflight를 별도 단계로 둡니다.

권장 명령 인터페이스는 다음과 같습니다.

1. 초안 생성: `generate-next-linear-algebra-post`
2. 게시: `publish-linear-algebra-post`

명령 이름 자체는 구현 시 조정할 수 있지만, `초안 생성`과 `게시`는 반드시 분리하는 것이 좋습니다.
